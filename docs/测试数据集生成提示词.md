# 隐私打码测试数据集生成提示词

## 项目背景

我们正在开发一款智能隐私打码助手，需要生成一批测试数据集来定义项目的验收标准。

### 目标用户

- **职场人士**：需要分享商务邮件、合同、方案等，打码需求高度个性化（如"隐藏我方报价但保留对方需求"、"打掉核心创意但保留背景说明"）
- **创业者/自由职业者**：分享项目讨论、客户沟通记录时，需要保护商业机密、客户信息、定价策略等主观隐私
- **电商卖家/客服人员**：需要批量处理订单截图、聊天记录，打码规则复杂（如"隐藏买家信息但保留订单号"、"打掉自己店铺名但保留商品信息"）
- **内容创作者**：分享素材、反馈、合作洽谈时，需要根据语义判断哪些是敏感的
- **隐私敏感用户**：日常分享聊天记录、快递单、证件照片等，打码需求依赖语义理解

这些用户的共同特点是：**打码需求高度个性化、依赖语义理解、无法用固定规则描述**。

### 核心痛点

传统AI工具只能识别固定格式的信息（身份证号、手机号等），无法：
- 区分"自己的名字需要打码，他人的名字不打码"
- 理解主观性强的隐私（如"方案创意"、"商业机密"、"内部评价"）
- 根据用户身份和场景做个性化判断

### 我们的解决方案

通过AI语义理解，支持基于身份、角色、场景的差异化打码策略。

---

## 任务说明

生成一批测试用例，每个用例包含：
1. **场景描述**：说明这是什么类型的文档、用户是谁、分享目的是什么
2. **输入文本**：完整的待打码文本内容（200-500字）
3. **期望输出**：标注结果，格式为 `[类型标签，文本内容，是否打码]`

---

## 核心测试场景

测试用例应该体现我们相比传统AI打码的优势，覆盖以下场景：

### 1. 同类信息的差异化处理
- 文本中有多个人名，需要区分己方/对方
- 文本中有多个电话/地址，需要区分私密/公开
- 例如：邮件中"发件人电话打码，客服热线不打码"

### 2. 主观性强的业务隐私
- 产品方案中的核心创意、设计思路
- 商务洽谈中的报价策略、成本结构
- 竞品分析中的对手弱点、市场数据
- 内部讨论中的人员评价、敏感话题

### 3. 基于角色和关系的判断
- 邮件中区分发件人/收件人信息
- 合同中区分甲方/乙方信息
- 聊天记录中区分双方身份

### 4. 语义关联的同一实体
- 同一个人的不同称呼（"张总"、"老张"、"张三"）
- 同一企业的不同表述（"我们公司"、"本公司"、"XX科技"）
- 同一内容的不同指代（"这个方案"、"我们的创意"、"上述设计"）

### 5. 隐私泄露链（信息组合风险）
- 单个信息不敏感，但组合后可定位具体个人
- 例如："某大学+计算机系+2020级+学生会主席"可锁定具体个人
- 需要识别这些组合并标注

## 输出格式

所有测试用例应输出为一个YAML文件，格式如下：

```yaml
test_cases:
  - id: 1
    name: "商务邮件-方案确认"
    description: "广告公司市场总监向客户发送方案确认邮件，需要分享给第三方征求意见。用户希望隐藏己方联系方式和核心创意，但保留客户信息以便第三方了解背景。"
    input_text: |
      [完整的待打码文本，200-500字]
    expected_output:
      - label: "发件人姓名"
        content: "王雷"
        should_mask: true
      - label: "联系电话"
        content: "+86 138 0000 0000"
        should_mask: true
      # ... 更多标注
    test_points:
      - "测试同类信息差异化：多个人名中区分己方和对方"
      - "测试语义关联：识别同一人的不同称呼"
      - "测试主观隐私识别：识别设计方案中的核心创意要素"
    why_traditional_fails: "传统工具只能识别所有人名和联系方式，无法区分身份和角色"
  
  - id: 2
    name: "第二个测试用例名称"
    # ... 后续测试用例
```

## 标签设计原则

**重要：使用最少的标签表现个性化划分，总共不超过8个标签类型。**

推荐的通用标签体系（可根据场景调整）：
1. **己方姓名** - 发件人/己方人员的姓名
2. **对方姓名** - 收件人/合作方人员的姓名
3. **己方联系方式** - 己方的电话、邮箱、地址
4. **对方联系方式** - 对方的电话、邮箱、地址（通常是公开信息）
5. **己方组织** - 己方的公司/组织名称
6. **对方组织** - 对方的公司/组织名称
7. **业务机密** - 创意、方案、报价、成本、战略等主观隐私
8. **内部信息** - 内部评价、决策过程、团队问题等敏感内容

**标注原则：**
- 同一类型的信息使用同一个标签（如所有己方人员都用"己方姓名"）
- 不要为每个具体内容创建新标签（如"发件人姓名"、"员工姓名"应统一为"己方姓名"）
- 业务机密可以包含多种内容（创意、报价、成本等），不需要细分
- 重点体现"己方 vs 对方"的差异化判断

## 生成要求

请生成 10-15 个测试用例，覆盖以下场景类型：

1. **商务邮件**（方案讨论、报价洽谈、合作确认）
2. **聊天记录**（工作群聊、客户沟通、内部讨论）
3. **合同文档**（甲乙方信息、条款细节）
4. **快递单/订单**（买卖双方信息）
5. **产品方案**（创意描述、竞品分析）
6. **内部文档**（人员评价、战略讨论）
7. **求职简历**（个人信息、工作经历）

每个用例应该：
- 文本真实自然，符合实际使用场景
- 明确测试某个核心能力
- 体现传统工具做不到的地方
- 标注结果准确、完整

## 使用说明

将此提示词提供给大语言模型（如GPT-4、Claude、Gemini等），让其生成完整的测试数据集。

**输出要求：**
- 生成一个完整的YAML文件
- 文件名：`privacy_masking_test_cases.yaml`
- 包含10-15个测试用例
- 每个用例严格遵循上述YAML格式

生成的数据集将用于：

1. **定义验收标准**：明确项目需要达到的识别能力
2. **评估模型性能**：测试AI Agent的识别准确率
3. **指导开发方向**：确保开发重点放在差异化功能上
4. **演示产品能力**：向用户展示我们相比传统工具的优势

---

## 注意事项

1. 生成的文本内容应避免使用真实的个人信息，使用虚构但真实的数据
2. 标注结果应该反映"合理用户"的打码需求，而非极端情况
3. 每个测试用例应该有明确的测试目标，避免重复
4. 优先覆盖传统工具做不到的场景，体现我们的差异化优势
