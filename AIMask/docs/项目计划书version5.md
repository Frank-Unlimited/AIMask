## 1 项目概述

### 1.1 项目背景

随着移动互联网的快速发展，用户每天产生和分享大量包含个人隐私信息的数字内容。根据中国互联网络信息中心发布的数据，我国手机网民规模已超过10亿，移动端图片和文档分享已成为日常生活和工作的重要组成部分。然而，这些分享内容中往往包含身份证号、手机号码、家庭住址、银行卡号、人脸图像等敏感隐私信息，一旦泄露将对用户造成严重的安全隐患。

目前用户在分享内容前进行隐私保护的主要方式仍是手动打码，这种方式存在以下问题：

| 问题类型 | 具体表现 | 影响程度 |
|---------|---------|---------|
| 操作效率低下 | 手动框选每个隐私信息，处理一张图需3-5分钟，10张图需30分钟以上 | 高 |
| 遗漏风险较高 | 边缘位置或不易察觉的敏感信息容易被忽略，特别是同一人的不同称呼 | 高 |
| 打码质量参差 | 过度打码破坏可读性，打码不足无法有效保护隐私 | 中 |
| 批量处理困难 | 多张图片需要重复操作，无法复用已有规则 | 高 |

效率对比如下：

| 场景 | 传统手动打码 | 本产品 | 效率提升 |
|-----|------------|-------|---------|
| 单张聊天记录 | 3分钟 | 30秒 | 6倍 |
| 10张快递单 | 30分钟 | 3分钟 | 10倍 |
| 复杂合同文档 | 5分钟 | 1分钟 | 5倍 |

市场竞品分析如下：

| 产品类型 | 代表产品 | 识别能力 | 智能程度 | 数据安全 |
|---------|---------|---------|---------|---------|
| 系统自带工具 | 手机相册编辑 | 无自动识别 | 低 | 本地处理 |
| 通用图片编辑 | 美图秀秀等 | 无自动识别 | 低 | 云端处理 |
| 专业打码工具 | 各类马赛克App | 简单格式匹配 | 中 | 多为云端 |
| 本项目 | 智能隐私打码助手 | AI语义识别+YOLO检测 | 高 | 端侧部署 |

综上所述，市场迫切需要一款能够智能识别多类型隐私信息、提供个性化打码建议、并保障数据处理安全性的移动端打码工具。

### 1.2 项目内容

本项目研发了一款面向安卓平台的智能隐私打码助手应用，主要内容包括：

- **隐私信息自动识别**：综合运用OCR文字识别、YOLO目标检测和大语言模型语义分析，自动定位图片和文档中的敏感信息区域，包括文字类（身份证号、手机号、姓名、地址等）和图像类（人脸、证件照片、二维码等）。
- **智能化打码建议**：基于三类Agent协同工作机制（A2A协议），系统自动判断文档类型、识别隐私分类、生成打码建议并附带判断理由，支持多轮交互式调整。用户可以随时了解AI的"思考过程"。
- **便捷交互体验**：实现点击即选词、语义关联打码、动态分类标签等创新交互方式，用户只需简单点击即可完成精准打码。
- **批量处理能力**：支持基于首张图片的交互式打码结果自动生成打码规则模板，一键应用到剩余所有文件。
- **数据安全保障**：采用端云协同架构，敏感数据在本地处理，AI能力云端提供，兼顾功能与安全。

| 核心能力 | 技术方案 |
|---------|---------|
| 隐私信息自动识别 | OCR字符级识别 + YOLO目标检测 + LLM语义分析 |
| 智能化打码建议 | 三类Agent协同工作（A2A协议），支持多轮交互，附带判断理由 |
| 便捷交互体验 | 点击即选词、语义关联打码、动态分类标签 |
| 批量处理能力 | 自动生成打码规则模板，一键应用到多文件 |
| 数据安全保障 | 端云协同架构，敏感数据本地处理 |

### 1.3 创新点

#### 1.3.1 技术创新点

本产品采用**OCR+YOLO+LLM三重技术融合**，实现了对文字、人脸、证件等多种类型隐私信息的智能识别。

**字符级OCR精准定位**采用实验室自主研发的OCR引擎，具备字符级识别能力，能够精确定位图片中每一个文字的位置，为后续精准打码提供支撑。

**DSPy自动提示词优化Agent模块**采用DSPy框架进行提示词自动优化，能够根据不同类型的文档自动调整识别策略，持续提升建议的准确度。

#### 1.3.2 功能创新点

##### 语义关联打码（业界首创）

传统打码工具只能识别和处理用户手动选中的内容，无法理解文本语义。本产品实现了业界首创的**语义关联打码功能**，能够理解同一实体在文档中的不同表述方式，实现一键关联打码。

- 理解"游总"、"老游"、"游四方"指向同一个人
- 理解"158开头的号码"和"尾号8888"可能是同一电话
- 理解"南京市鼓楼区"和"鼓楼区南京"是同一地点
- 用户确认后可一键批量打码所有关联内容，不会误伤无关内容

##### 点击即选词

传统打码需要用户精确框选区域，操作门槛高且容易选不准。本产品基于字符级OCR识别结果，用户只需点击文字区域，系统自动选中对应的完整文本单元。

- 点击"138"中的任意数字，自动选中包含该数字的完整手机号
- 点击地址中的"鼓楼"，自动选中完整地址文本块
- 配合AI语义分析，可识别同一实体的关联内容并支持联动选择

##### 隐私泄露链检测

单个信息可能不敏感，但多个信息组合起来可能定位到具体的人。本产品实现了**隐私泄露链检测功能**，能够识别信息组合带来的身份定位风险。

| 单条信息 | 风险等级 | 组合后风险 |
|---------|---------|-----------|
| "某大学" | 低 | - |
| "计算机系" | 低 | - |
| "2020级" | 低 | - |
| "学生会主席" | 低 | - |
| 以上组合 | - | 高：可锁定具体个人 |

##### 动态分类标签与色彩标识系统

系统根据识别到的内容自动生成分类标签，并采用分类色彩标识机制——不同类别的隐私信息使用不同颜色的遮罩框进行可视化区分，让用户一目了然：

| 隐私类别 | 颜色标识 | 交互方式 |
|---------|---------|---------|
| 姓名/称呼 | 蓝色遮罩 | 点击标签可批量选中/取消 |
| 电话号码 | 绿色遮罩 | 点击标签可批量选中/取消 |
| 地址信息 | 橙色遮罩 | 点击标签可批量选中/取消 |
| 业务隐私 | 紫色遮罩 | AI识别后动态显示 |
| 关联方 | 红色遮罩 | AI识别后动态显示 |

这种设计的优势在于：

- **直观区分**：用户通过颜色即可快速识别不同类型的隐私信息
- **批量操作**：点击界面下方的分类标签按钮，可一键全选/取消该类型的所有打码
- **动态扩展**：标签根据文档内容动态生成，而非固定模板，AI可根据语义识别新增分类

#### 1.3.3 其他创新点

传统打码工具遵循单向模式：用户是"调用者"，工具是被动执行命令的"工具"。用户必须逐一检查每个识别结果，承担全部决策压力。本产品颠覆性地将用户与AI的关系定位为平等的**"协作伙伴（Cowork）"**——AI不再是等待指令的工具，而是会主动思考、主动解释、主动询问的合作者。

| 维度 | 传统工具模式 | 本产品Cowork模式 |
|-----|------------|-----------------|
| 关系定位 | 用户调用 → 工具执行 | 用户 ↔ AI 协作对话 |
| 交互方式 | 用户逐一检查AI结果 | AI主动解释，用户选择性响应 |
| 决策压力 | 全部由用户承担 | AI提供建议和理由，用户决定采纳与否 |
| AI角色 | 被动执行命令 | 主动思考、解释、询问 |
| 信任建立 | 无（工具无需信任） | 通过透明化逐步建立 |

在整个打码过程中，AI会主动解释自己的判断逻辑——为什么认为这里需要打码、依据是什么、属于哪个隐私类别。用户可以随时了解AI的"思考过程"，双方通过持续交流不断对齐打码的颗粒度和偏好，最终形成默契配合。

用户对AI的信任不是天然存在的，而是通过交互逐步建立的。我们深刻认识到：用户第一次使用时，只会把这个当成普通打码工具来用。但随着与AI的交互增多，用户会逐渐发现AI的判断是靠谱的、AI的解释是有道理的，从而愿意给予AI更多的信任和自主权。

这一信任建立过程体现为三种协作模式的渐进切换：

| 协作模式 | 适用阶段 | 用户心态 | AI行为 | 交互深度 |
|---------|---------|---------|-------|---------|
| 快速模式 | 初次使用 | "我只是需要一个打码工具" | 自动识别并标记隐私内容，用户直接确认 | 低 |
| 协作模式 | 建立信任后 | "AI的建议挺靠谱的" | 遇到不确定时主动询问，展示判断依据 | 中 |
| 教学模式 | 深度信任后 | "我想让AI完全理解我的需求" | 详细解释思考过程，主动请求反馈 | 高 |

为了让信任自然建立而非强迫用户参与，我们设计了"可忽视的协作"机制——AI主动但不纠缠，提供价值但不打扰：

- AI的提问和建议都可以被忽略，系统会按默认方式处理
- 用户不需要回答每一个问题，AI会自动推断
- 随着用户偶尔的响应，AI逐渐理解用户偏好
- 用户可以随时切换协作模式，AI会记住偏好

**核心交互循环如下：**

1. 用户上传图片，AI自动完成初步打码
2. 用户点击某个元素或输入自然语言指令，AI弹窗展示打码建议、给出理由、询问是否确认
3. 用户确认或调整
4. 用户点击已打码内容，AI询问是否撤销并说明影响
5. 重复2-4步，经过2-4轮交互后生成最终结果

这种模式的核心价值在于：用户从"被动检查AI结果"转变为"与AI对话协商"，信任在无感知中逐步建立，最终实现人机默契配合。

系统通过**mem0记忆框架**，持续记录用户的每次操作（点击、确认、撤销、忽略），构建个性化的用户画像。这不仅包括用户的打码偏好（如"公司电话不打码"），还包括用户的信任程度和协作模式偏好，实现"越用越懂你"的效果：

| 处理阶段 | AI准确率 | 用户调整次数 | 说明 |
|---------|---------|------------|------|
| 第1张图 | 85% | 约10处 | 初始状态，使用通用规则 |
| 第3张图 | 92% | 约5处 | 学习用户偏好，识别更准确 |
| 第5张图 | 95% | 约2处 | 深度学习，几乎无需调整 |
| 第10张图 | 98% | 仅确认 | 高度精准，用户只需一键确认 |

与传统机器学习"用户行为→模型学习"的单向模式不同，本产品强调人和AI共同建立打码规则：

1. 遇到新文档类型时，AI会询问用户如何处理各类信息
2. 用户可以通过例子教AI理解业务特定的隐私概念
3. AI会主动展示学到的规则，用户可以确认或修正
4. 最终形成的是"人和AI共同制定的打码策略"，而非"AI单方面学习的模型"

系统能够根据用户在首张图片上的交互操作，自动总结出打码规则（包含用户偏好、打码示例等），并将规则应用到剩余的所有图片上，大幅提升批量处理效率。在批量处理过程中，AI会主动识别不确定的内容并询问用户，而非盲目套用规则。

## 2 技术方案

### 2.1 总体架构

本系统采用前后端分离的微服务架构，由安卓前端、OCR服务、YOLO检测服务和Agent服务四个核心模块组成。有着以下四点优势：

- **高度解耦**：各模块独立部署，便于单独升级和扩展
- **弹性扩展**：可根据负载动态调整各服务资源
- **技术灵活**：各模块可采用最适合的技术栈
- **端云协同**：敏感数据在端侧处理，AI能力云端提供

系统分为：

- **客户端层（安卓前端应用）**：包含图片上传/放大缩小、Canvas打码渲染、字符级可点击框、分类标签动态展示、批量导出等模块
- **服务层**：通过HTTP API (JSON)与客户端通信，包含OCR服务（PaddleOCR，负责字符级识别、坐标定位、置信度输出）、YOLO服务（YOLOv8，负责人脸检测、证件检测、二维码检测）、Agent服务（LangChain，包含初始打码Agent、迭代打码Agent、批量处理Agent）

| 模块 | 技术选型 | 开源协议 | 选型理由 |
|-----|---------|---------|---------|
| 安卓前端 | Kotlin + Jetpack Compose | Apache 2.0 | Google官方推荐，现代化UI框架 |
| 图像渲染 | Android Canvas API | Apache 2.0 | 原生高性能图形API |
| OCR引擎 | 实验室自研 | - | 实验室自研，准确率高 |
| 目标检测 | YOLOv8 | AGPL-3.0 | 业界最先进的实时检测模型 |
| 智能Agent | 多模型支持（GPT-4/Gemini/Qwen等） | - | 统一LLM接口，支持灵活切换 |
| 后端服务 | FastAPI | MIT | 高性能Python Web框架 |

Agent框架设计上支持多种大语言模型（包括GPT-4、Gemini、Claude、Qwen等），通过统一的LLM接口层实现模型的灵活切换。架构设计参考Google ADK理念，可深度集成Google生态（如Firestore存储用户画像、MLflow进行模型监控），同时也兼容其他云服务。框架设计简洁，符合"Keep it Simple"的工程理念，实际部署时可根据需求选择最适合的大模型。

### 2.2 功能详述

#### 2.2.1 图片上传与预处理

- 支持从相册选择、文件管理器选择、拍照上传三种方式
- 支持格式：JPG、PNG、PDF
- 支持单张模式和批量模式

#### 2.2.2 OCR字符级识别

系统采用实验室自主研发的OCR引擎，具备字符级识别能力。识别结果包含：

- 图片中的每一个字符
- 每个字符在图片上的精确位置坐标
- 整行文字的内容，方便AI理解上下文语义

#### 2.2.3 YOLO目标检测

检测图片中的非文字类隐私信息：

| 类别 | 检测对象 | 准确率 |
|-----|---------|-------|
| face | 人脸 | 98%+ |
| id_card | 身份证 | 96%+ |
| bank_card | 银行卡 | 95%+ |
| qrcode | 二维码 | 99%+ |
| barcode | 条形码 | 99%+ |

#### 2.2.4 智能Agent打码建议

系统采用多Agent协同架构，通过A2A（Agent-to-Agent）协议实现不同场景Agent的协作，便于后续扩展更多场景。

- **初始打码Agent**：判断文档类型（聊天记录、合同、快递单等）、识别文档中的隐私信息分类、生成初步打码建议列表，并附带判断理由。
- **迭代打码Agent**：理解用户点击位置的意图，推测用户想要打码的完整词语边界，弹窗展示建议内容、判断依据和隐私分类，询问用户是否确认，识别与当前选择相关联的其他内容。
- **批量处理Agent**：根据用户交互记录和mem0中的用户画像生成打码规则，将规则应用到新图片，遇到不确定内容时主动询问

#### 2.2.5 交互式打码

**主界面布局说明**，界面从上到下依次为：

- 顶部导航栏：返回按钮、当前图片序号（如"图片1/8"）、批量应用按钮、预览对比按钮
- 图片显示区域：支持缩放、拖动，显示带可交互文字框/遮罩的图片
- 动态分类标签区：如 [姓名(3)] [电话(2)] [地址(1)]
- 自然语言输入框：可输入如"把所有人名都打掉"的指令，右侧有语音按钮
- 操作按钮区：[撤销] [进一步打码] [完成编辑]

**核心交互方式分为：**

- 点击添加打码：点击图片上任意未高亮文字，AI弹窗展示打码建议、判断理由和所属分类，询问用户是否确认
- 点击取消打码：点击已高亮区域，取消该部分打码
- 框选多个文字：长按拖动，框选想要的范围
- 自然语言指令：输入"把所有人名都打掉"等指令
- 分类标签批量操作：点击标签可全选/取消该类型的所有打码（不同类别使用不同颜色遮罩区分）

### 2.3 关键技术

#### 2.3.1 字符级OCR技术

- **核心引擎**：实验室自研
- **数据流程**：输入图片 → 图像预处理(降噪/增强) → 文字检测(DB算法) → 文字识别(CRNN) → 结构化输出
- **字符级精度**：精确到单个字符，支持任意粒度的打码
- **坐标精准**：四角坐标定位，打码边界清晰
- **置信度输出**：识别结果带置信度，支持质量筛选

#### 2.3.2 YOLO目标检测技术

- **核心模型**：YOLOv8（Ultralytics开源，AGPL-3.0协议）
- **推理框架**：ONNX Runtime / TensorRT
- **部署方式**：FastAPI封装为RESTful服务
- **实时检测**：单张图片处理时间<100ms
- **高精度**：人脸检测准确率98%+
- **多目标**：单次推理可检测多种目标类型
- **端侧可部署**：支持NCNN/TensorRT加速

#### 2.3.3 LLM语义理解与Agent技术

- **Agent框架**：基于FastAPI的自定义Agent服务，参考Google ADK设计理念
- **大语言模型**：支持多模型切换（GPT-4/Gemini/Claude/Qwen等）
- **Prompt优化**：支持DSPy自动优化框架，可根据输入输出样例优化提示词
- **记忆机制**：mem0用户画像与偏好记忆（支持跨会话学习）
- **数据存储**：支持Firestore等云数据库（存储用户画像和交互历史）
- **模型监控**：支持MLflow集成（监控Agent性能和响应质量）
- **语义理解**：基于大语言模型的深度语义分析能力
- **意图推测**：准确理解用户点击背后的真实意图
- **关联识别**：识别同一实体的不同表述方式
- **理由生成**：每次建议都附带判断依据，让用户知道AI在想什么
- **画像构建**：通过mem0持续学习用户特征和打码习惯
- **规则生成**：自动将用户偏好转化为可复用规则

采用A2A（Agent-to-Agent）协议实现多Agent协同，各Agent职责明确、松耦合，便于后续增加新的场景Agent（如视频打码Agent、文档批注Agent等）。

#### 2.4.2 UML设计

**用例说明**：用户可执行的核心操作包括：上传图片、查看AI打码建议、点击选择、输入自然语言、批量处理、导出结果

**交互序列说明**：

1. 用户上传图片到前端App
2. 前端App向OCR服务发送识别请求，获取字符列表
3. 前端App向YOLO服务发送检测请求，获取检测结果
4. 前端App向Agent服务发送初始分析请求，获取打码建议
5. 前端显示建议给用户
6. 用户点击选择后，前端向Agent服务发送迭代请求，获取推测结果
7. 用户确认打码后导出

**状态流转说明**：空闲 → 上传图片 → 识别中 → OCR+YOLO完成 → AI分析 → Agent返回建议 → 编辑中 → 用户点击 → 推测中 → 编辑中 → 完成 → 批量处理/导出

## 3 项目计划

### 3.1 可行性分析

#### 3.1.1 技术可行性分析

本项目所依赖的核心技术均已相当成熟。在OCR领域，PaddleOCR和Tesseract等开源方案已能达到95%以上的识别准确率；在目标检测领域，YOLOv8作为业界最先进的实时检测模型，mAP可达96%以上；在大语言模型领域，GPT-4、Claude以及国产大模型均提供稳定的API服务，响应时间控制在秒级；在安卓开发领域，团队成员具备Kotlin和Jetpack Compose的开发经验，技术储备充足。

技术层面的主要风险在于OCR识别准确率和LLM响应延迟。针对OCR可能出现的识别偏差，我们采用字符级识别并支持用户手动调整；针对LLM响应延迟问题，我们在前端设计了加载动画，并在后端实现常用规则的本地缓存，以提升用户体验。

#### 3.1.2 资源可行性分析
